PDF-Analyse mit Streamlit-Interface, deployed in Docker.

Projekt 3 — Preis-Monitor & API
Scrapy + Playwright scrapen Preise, FastAPI gibt Daten aus, tenacity handelt Rate Limits.
ScrapyPlaywrightFastAPItenacity
Was du baust
  • Scrapy Spider crawlt statische Produktseiten (z.B. MediaMarkt oder idealo) — Produktname, Preis, Verfügbarkeit
  • Playwright übernimmt wenn JavaScript-Rendering nötig ist (dynamisch geladene Preise)
  • SQLite-Datenbank speichert historische Preise mit Zeitstempel

  • FastAPI exponiert zwei Endpoints: GET /prices/{produkt} und GET /history/{produkt}

  • Optional: E-Mail-Benachrichtigung wenn Preis unter Schwellenwert fällt
Warum dieses Projekt
  • Zeigt den kompletten Produktions-Stack: Scraping → DB → API → Error Handling
  • Hat einen echten Nutzen — du kannst es selbst verwenden
  • Direkt verkaufbar an KMUs die Konkurrenzpreise monitoren wollen
Ziel: Ein produktionsreifer Service der zuverlässig läuft, auch wenn externe Seiten langsam oder down sind.

Projekt H1 — Hase-Intrusion-Detektor
ESP32-CAM filmt live → Raspberry Pi erkennt Hasen per YOLOv8 → Push-Benachrichtigung aufs Handy.
OpenCVYOLOv8Tracking ESP32MQTTPush StreamlitFastAPI
Architektur — 3 Schichten
  • ESP32-CAM → sendet MJPEG-Stream per HTTP an lokale IP des Raspberry Pi
  • Raspberry Pi 4 → empfängt Stream, verarbeitet Frames mit OpenCV + YOLOv8
  • Streamlit Dashboard → hosted auf Pi, zeigt Live-Feed mit Bounding Boxes und Alarm-Log
ESP32-CAM Setup
  • Arduino IDE, CameraWebServer-Sketch flashen — WLAN eintragen, fertig
  • ESP32 streamt unter http://[IP]/stream als MJPEG
  • cv2.VideoCapture("http://[ESP32-IP]/stream") liest den Stream auf dem Pi
Detection Pipeline auf dem Pi
  • YOLOv8n (nano, ~4MB) via Ultralytics — ca. 5–10 FPS auf Pi CPU, reicht für Garten
  • Nur Klasse "rabbit" triggert Alarm — Alarm-Zone als Polygon in OpenCV definiert
  • ByteTrack verfolgt Entitäten über Frames, gibt jeder eine persistente ID
  • ntfy.sh Push-Notification mit Frame-Snapshot wenn Hase 3+ Sekunden in Zone
Hardware (~40€)
  • ESP32-CAM AI-Thinker: ~8€ · FTDI Serial-Adapter: ~4€ · MicroSD 32GB: ~6€
  • Raspberry Pi 4 (2GB): bereits vorhanden oder ~45€ · USB-C Netzteil: ~8€
Basis-Schritt: Ein System das wirklich funktioniert. Wächst direkt zum Security-System unten.
Projekt H2 — Multi-Camera Security System
Mehrere ESP32-Cams → je ein Docker-Container mit YOLOv8 → Redis Pub/Sub → zentrales Überwachungs-UI mit Person Detection.
OpenCVYOLOv8Redis WebSocketsDocker ComposeNGINX FastAPIStreamlit
Systemarchitektur
  • ESP32-CAM-1,2,3 → senden MJPEG-Streams ins lokale Netzwerk
  • Detection-Container pro Kamera → liest Stream, läuft YOLOv8, publiziert Events auf Redis
  • Redis Pub/Sub → Nachrichtenbus zwischen Containern, kein direktes Polling nötig
  • FastAPI Central API → subscribt Redis, speichert Events in SQLite, exponiert WebSocket-Endpoint
  • Streamlit UI → empfängt Updates via WebSocket, zeigt alle Kamera-Feeds live
  • NGINX → reverse proxy, routet /cam1, /cam2, /api, /ui unter einer einzigen IP
  • Alles in docker-compose.yml: ein Befehl startet das gesamte System
docker-compose.yml Struktur
  • service: redis — redis:7-alpine, kein Config nötig
  • service: mosquitto — MQTT-Broker für ESP32-Kommunikation
  • service: cam1-detector — build: ./detector, env: CAM_URL=http://192.168.1.10/stream
  • service: cam2-detector — selbes Image, anderer CAM_URL-Parameter
  • service: camN-detector — skaliert durch env-Variable, kein neues Image
  • service: api — FastAPI, depends_on: redis, exponiert /ws WebSocket-Endpoint
  • service: ui — Streamlit, verbindet zu /ws für Live-Updates
  • service: nginx — nginx:alpine, routet alle Services unter Port 80
Detection-Container (wiederverwendbar)
  • Ein einziges Docker-Image: liest CAM_URL aus Umgebungsvariable
  • Öffnet cv2.VideoCapture(os.getenv("CAM_URL")), läuft YOLOv8 auf jedem Frame
  • Person erkannt → redis.publish("detections", json.dumps({cam_id, bbox, confidence, timestamp, frame_b64}))
  • Neue Kamera hinzufügen = eine neue Service-Zeile in docker-compose.yml, sonst nichts
FastAPI WebSocket Hub
  • asyncio + aioredis subscribt alle Redis-Channels
  • Eingehendes Event → sofort an alle verbundenen WebSocket-Clients weitergeleitet
  • SQLite speichert alle Ereignisse persistent: Kamera-ID, Zeitstempel, Konfidenz, Frame-Pfad
  • REST-Endpoints für Alarm-History, Kamera-Status, Statistiken
Streamlit Monitoring UI
  • Grid-Layout: N × M Kacheln, eine pro Kamera, Live-Feed mit Bounding Boxes
  • Echtzeit-Updates via WebSocket: neue Detektion → Kachel blinkt, Alert-Banner erscheint
  • Alarm-Log mit Filter nach Kamera, Zeitraum, Konfidenz-Schwellenwert
  • Kamera-Management: Streams hinzufügen/entfernen ohne System-Neustart
  • Statistik-Charts: Detektionen pro Stunde, Hotspot-Heatmap pro Kamera
Warum dieses Projekt heraussticht
  • Echte Microservices-Architektur — kein Tutorial-Projekt sondern ein System das so in Produktion aussieht
  • Horizontale Skalierbarkeit: neue Kamera = eine Zeile YAML, kein Code
  • Verbindet Hardware, Computer Vision, Backend, Real-Time Web in einem System
  • Funktioniert komplett offline im Heimnetz — kein Cloud-Abo, keine laufenden Kosten
  • Custom YOLO Training auf eigenen Person-Daten als natürliche Erweiterung
Ziel: Ein produktionsähnliches System das zeigt dass du Architekturentscheidungen treffen und umsetzen kannst — nicht nur Tutorials nachbauen.

Projekt 8 — RAG Document QA
Dokumente embedden, per LLM befragen, instructor für strukturierte Outputs, RAGAS-Evaluation.
RAGinstructorRAGAS
Was du baust
  • User lädt PDFs hoch (z.B. wissenschaftliche Paper, Geschäftsberichte)
  • pdfplumber extrahiert Text, RecursiveCharacterTextSplitter chunked in 512-Token-Blöcke mit 50 Token Overlap
  • HuggingFace Embeddings (all-MiniLM-L6-v2) konvertieren Chunks zu Vektoren
  • ChromaDB speichert die Vektoren lokal persistent
  • Bei einer Frage: Top-5 relevanteste Chunks werden retrieved, per LLM beantwortet
  • instructor erzwingt strukturierten Output: {answer: str, sources: list, confidence: float}
  • RAGAS berechnet automatisch: Context Recall, Faithfulness, Answer Relevance auf 50 Testfragen
  • Streamlit-Interface mit Quellenangaben unter jeder Antwort
Ziel: Verstehen warum RAG-Systeme gut oder schlecht sind — und es messen können.
